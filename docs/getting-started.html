Getting started

No experience needed. This page explains, in plain English, everything you'll touch.

🚦

This guide picks up after Mac Launchpad setup has finished installing. If a command ever says command not found, setup may not be complete — open the Help page, or ask an assistant to “run the doctor check.”

1What is the “Terminal”?

The Terminal is a plain text window where you type a command (an instruction for the computer), press Enter, and the Mac does it. That's it. To open it: press ⌘ (the Command key, just left of the spacebar) + Space — that opens Spotlight, the Mac's search box — then type Terminal and press Enter.

One convention for these guides: a monospaced word like claude is something you type into the Terminal; a “sentence in quotes” is something you say to an AI assistant. Hit a word you don't know? There's a plain-English glossary at the bottom of this page.

When the Terminal is ready for you, it shows a prompt — a short line (often ending in a %) with a blinking cursor. That's just the computer saying "your turn — type something." You can't break anything by looking at it; the Terminal only does what you actually type and Enter.

⚠️

Watch out for one thing: if you paste a command and something explains it back to you or offers to “inspect it,” you've landed in macOS's “Compose” / Writing Tools pop-up (Apple Intelligence) or an AI chatbot — not the Terminal. The real Terminal never talks back; it just runs what you type. Close the pop-up and paste into the Terminal window instead.

You'll also have a nicer-looking terminal called Alacritty after setup — it works exactly the same, just prettier. Use either.

💡

Here's the good news: you mostly won't type commands yourself. You'll ask the AI assistants in plain English, and they run the commands for you. Why did the Terminal make a great waiter? You ask for something, and it just brings it.

2Your three AI assistants

Setup installs three AI coding assistants. All live in the Terminal and all can build whole apps for you by chatting in plain English — use whichever you like, or get a second (and third) opinion:

Claude Code

Type claude. Runs your full setup; a great all-rounder.

Codex

Type codex. From OpenAI. A handy second opinion.

Antigravity

Type agy. From Google. Strong at running several tasks in parallel and switching AI models on the fly.

All three share the same house rules, so they behave consistently and explain things as they go.

Inside Antigravity, type /model to switch the underlying AI model (Gemini, Claude, GPT…), /usage to check your quota, and /logout to sign out. The first time you run agy it opens a Google sign-in (in Chrome) — and may ask you to pick a colour theme; choose a dark one.

Prefer a window over the terminal? Antigravity also has a desktop app at antigravity.google/download (opens in a new tab) — it shares the same engine and your sessions. This setup standardises on the agy command line, but the app is there if you want it.

🔑

The first run signs you in (once). The first time you type claude, codex, or agy, each opens a one-time login in your browser — your Claude account, your ChatGPT account for Codex, and a Google account for Antigravity. Your private backups also need a one-time GitHub sign-in: run gh auth login, or just ask an assistant to “sign me in to GitHub.”

One more distinction: commands that start with a slash (like /exit, /model) are typed inside an assistant, not at the plain Terminal line.

3Starting an assistant

Every project lives in its own folder — exactly like a folder on your desktop, just a place to keep one project's files together. Your project folders live inside a folder called Developer in your home folder (your account's main folder, which the Terminal writes as ~ — a shorthand for "home"). So ~/Developer/my-idea just means "the my-idea folder, inside Developer, inside home."

To make a fresh, safe project folder and step into it, type this command in the Terminal:

Type in Terminal, then Enter 
mkproj my-idea

That creates ~/Developer/my-idea, sets up a “save point” system (called Git — more on that below), a safety check that stops you accidentally sharing passwords, and — once you're signed in to GitHub — a private online backup, then moves you into the folder. (“Move into the folder” = cd, short for “change directory.” A directory is just another word for a folder. mkproj does the cd for you.)

Now start an assistant — type its name and press Enter:

Type in Terminal, then Enter 
claude

You're now chatting with Claude inside that folder, so anything it builds lands there. To leave and return to the Terminal, type /exit or press Ctrl (the Control key) + C twice — that works to exit all three assistants.

🧱

Want to start from something that already runs? launchpad new scaffolds a ready-made website, phone app, or game — already set up with tests and the same safety net — then you just tell the assistant how to change it.

4Asking for a build

Once an assistant is running, you talk to it in plain English — no commands, no code. (Quick rule of thumb for this whole site: monospaced words like claude are typed into the Terminal; “sentences in quotes” are things you say to the assistant.) Just describe what you want, like you'd tell a person you hired. Be specific about the goal, not the how. Examples that work well:

A website
“Build me a one-page site for my bakery with a menu, photos, and a contact form. Make it look warm and modern.”
A small app
“Make a habit tracker where I check off habits each day and see a streak. Save my data on my computer.”
A game
“Create a simple browser game where I move a spaceship and dodge asteroids. Keep score.”

The assistant will explain its plan, build it step by step, test it, and show you the result. If something's not right, just say so — “make the button blue,” “that's too small,” “add a dark mode.”

🧭

Expect a plan first. Your assistants follow a disciplined workflow (the “Superpowers” framework): for anything non-trivial they'll ask a few clarifying questions, sketch a short plan, write tests, and review their own work before calling it done. So the first reply to “build X” may be questions, not code — that's intentional, and it produces much better results. They also have skills for polished visual design and for making real documents (PDF, Word, slides, spreadsheets).

🧪

They test their own work. A test is a tiny program that automatically checks the app still works. Your assistants write and run several kinds (checking the logic, clicking through the whole app, accessibility, and how it looks) and can drive a real browser themselves — so what they hand you actually works. You don't have to do anything. If you're ever curious, inside a project the assistant built you can run npm test yourself (npm is the tool that runs an app's code libraries), and every push to GitHub re-runs the tests automatically.

✅

Want to see this end-to-end first? Follow the Build your first app walkthrough.

5Saving your work (“commit”)

A commit is a save point you can always return to. The assistants commit automatically at milestones, but you can ask any time: “save this as a checkpoint.” If a change ever makes things worse, you can say “undo the last change” and go back.

This is why every project starts with Git — think of it as an unlimited undo button (a time machine) for your whole project. Each commit is a snapshot you can jump back to. Why are programmers great at relationships? They commit early and commit often.

☁️

Your work is backed up automatically. Each project is also pushed to a private repository on GitHub — a real off-machine backup (a single Mac is not a backup). It's private by default, so your code is never exposed by accident. Ask an assistant to “push my latest work” any time.

🔔

If you ever leave changes unsaved or unpushed, you'll get a one-line reminder when you open that project. And if your AI spending suddenly spikes, you'll get a heads-up — check anytime with launchpad spend.

6Putting it online

When you're proud of something, ask the assistant to publish it: “put this on the internet so I can share a link.” For web apps it'll typically use Vercel (free to start) and hand you a URL. For code, it can push to GitHub — your free, online backup and home for projects.

For an instant share — a quick page, an image, a PDF — all three assistants also have here.now built in. Just say “publish this to here.now” and you get a live link in seconds, no account needed (anonymous links last 24 hours; sign up for permanent ones, custom domains, and private cloud “Drives”).

🌍

here.now links are public by default — anyone with the link can see what you publish. Don't put private or sensitive files on an anonymous link; ask for a password-protected or restricted site instead.

🔒

Never paste passwords or secret keys directly into your code. If an app needs one, the assistant keeps it in a private file called .env.local — Git is set to ignore that file, so it's never backed up or published. If you're ever asked to share a “secret key,” double-check first.

The rest below are tools you'll meet later — good to know they exist, but you don't need any of them to publish your first project. Skim and move on.

🔑

Using 1Password? You can keep API keys there instead of a file: launchpad secrets set NAME to store one, and launchpad secrets run -- npm run dev to run your app with keys injected in memory. It's optional — without it, keys live safely in a git-ignored .env.local.

📦

You already have Docker (via an app called OrbStack) — the standard way to package apps and run services like databases. You rarely touch it directly: when your app needs a database, the assistants start one for you. To see what's running, open the OrbStack app — it lists your containers, shows their logs, and even lets you browse their files. To put a containerised app online, the assistants can use fly launch.

🐞

If your app breaks, the assistants can see why. New apps come pre-wired for Sentry — a free service that captures errors with the exact file and line. All three assistants can read those errors and fix them: just say “check Sentry and fix the error.” Turn it on by adding a free Sentry key when you're ready — an assistant will walk you through it.

🗄️

See your data. When your app stores information in a database, open Beekeeper Studio to look at it and edit it in a friendly window — no database commands needed.

Plain-English glossary

Every field has its own slang. Here's the jargon you'll bump into, in everyday words. You don't need to memorise any of it — skim it once, and come back when a word looks unfamiliar.

app
Short for “application” — any piece of software you use, like a website, a phone app, or a game.
command
A short instruction you type into the Terminal and run by pressing Enter (e.g. claude).
prompt
Two meanings: the blinking line where the Terminal waits for your next command; and the plain-English request you type to an AI assistant (“build me a…”). Context makes it clear which.
folder / directory
The same thing — a labelled box that holds related files. Your projects each get their own.
Git
An automatic “save point” system. Every commit is a snapshot you can return to — an unlimited undo button for a whole project.
repo
Short for “repository” — the folder Git is tracking, i.e. one project's files plus its full history of save points.
GitHub
A website that stores your repos online — your off-machine backup and where code lives so others (or future you) can grab it. Private by default here.
server
A program that hands out a website or app when something asks for it. A local server runs on your own Mac just for you while you build.
localhost
A nickname for “this computer.” A link like http://localhost:5173 opens the app running on your Mac — only you can see it.
deploy
Put your app on the internet so other people can open it at a real web address. (“Publish” means the same thing.)
API key
A secret password that lets your app talk to another service (like payments or email). Treat it like a house key — never paste it into public code.
database
Where an app stores information so it sticks around — think of a smart spreadsheet your app reads from and writes to.
bug
A mistake in the code that makes something behave wrong. Fixing one is “debugging.” (Named after a real moth found in a computer in 1947 — true story.)
🎓

That's the jargon decoded — you now speak enough Developer to follow along. Why don't programmers like nature? Too many bugs.

Stuck?

Two things fix almost everything: ask the assistant “something went wrong — can you fix it?” (paste any red text you see), or check the troubleshooting page.

Need to send your setup to someone for help? Run launchpad report — it bundles everything useful into one secret-free file (it scans itself for keys first) that you can share.
