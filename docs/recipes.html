Recipes

The hardest part of building isn't the tools — it's knowing what to say. This page is a cookbook of copy-paste prompts. Find what you want to make or do, copy it, paste it to an assistant, and tweak the words. You don't write any code.

🧭

Your assistants plan before they build (the “Superpowers” workflow), so the first reply is often a few clarifying questions. That's normal and a good sign — answer them and you'll get much better results.

📋

How to read this page. Brand new? Do the Get started guide and your first app first. Each box below has a Copy button in its corner. There are two kinds of box:

  • Type to Claude Blue boxes are plain-English things you say to an assistant in the chat. Change the words freely — they're starting points.
  • Run in Terminal Dark boxes are commands you type into the Terminal. (Anything after a # is just a note — it isn't run.)

Why did the recipe make a great programmer? It always knew what to say at run-time.

⭐ The golden rule: one project, one folder

Before any recipe below, here's the single most important habit — and the one beginners most often skip: give every new idea its own folder inside ~/Developer, and never build two different projects in the same folder.

The easiest way is one of these two commands. Both create the folder and set up its safety net for you:

Run in Terminal 
mkproj my-idea       # makes ~/Developer/my-idea and moves you into it
# — or, to start from a ready-made template (web / phone / game) —
launchpad new        # asks what you're building, then sets it all up

Why this matters so much when you're new:

Nothing gets tangled
Each project's files stay separate. Two apps sharing one folder will overwrite each other's settings — a classic, confusing mess that's hard to undo.
Your undo button works per project
mkproj and launchpad new turn on Git (an automatic save-history) for that folder. You can roll one project back to yesterday without touching the others.
Each gets its own private backup
Every project becomes its own private GitHub repo, so your work is safe off your Mac — and one project's history never mixes into another's.
The assistant stays focused
When you run claude inside a project folder, it works on that app. Start it in your home folder or a folder full of projects and it gets confused about what you mean.
You can always find it
Everything lives in one place: ~/Developer. Type dev to jump there, then cd the-folder-name to go into one.
💡

Coming back to a project later? Don't start a new folder — go back into the one you already have: type dev (jumps to ~/Developer), then cd my-idea, then claude. You're right where you left off, with all your history intact.

Start something new

Each of these begins a brand-new project — so start it in its own folder (the golden rule above). Pick what you'd like to make:

📄

A web page

A site people look at — a portfolio, a shop page, an event invite.
🧩

A web app

A site people use — a to-do list, a tracker, a calculator.
📱

A phone app

A real app on your phone, previewed live as you build.
🎮

A game

A browser game, or a real 2D game with Godot.

📄 Web pages

A site to show the world — mostly something people read and look at. Start the folder (mkproj my-site then claude), then say one of these:

Type to Claude, then Enter 
Build a warm, modern one-page website for Rosa's Bakery: a hero with the
name and a tagline, a short "about us", a menu of 4-5 items with prices,
opening hours, and a contact line. Make it look great on a phone too, then
start it and show me.
Type to Claude, then Enter 
Make me a personal portfolio site: a friendly intro with my name and what
I do, a few project cards with images, and a contact button. Clean and
professional. Use placeholder text and images I can swap later.
Type to Claude, then Enter 
Build a simple invite page for my daughter's birthday party: the date,
time, address, an RSVP button that emails me, and a cheerful design with
balloons. Mobile-friendly.

Type to Claude, then Enter 
Build a to-do list app: I add tasks, check them off, and delete them.

Write a couple of tests and show it running in my browser.
Type to Claude, then Enter 
Make a habit tracker: I add habits, tick them off each day, and see a

Build a simple budget tracker: I add income and expenses with a category,

📱 Phone apps

📲

One free download first: get the Expo Go app on your phone (App Store on iPhone, Google Play on Android). It's what opens your project on your phone while you build.

Start a phone project with launchpad new → choose mobile, then claude. Describe the app:

Type to Claude, then Enter 
Build a simple tip calculator: I type the bill, tap a tip percentage
(15/18/20%), and it shows the tip and total. Big, easy-to-tap buttons.
Then start it so I can preview it on my phone.
Type to Claude, then Enter 
Make a water-reminder app: I set a daily goal, tap a button each time I
drink a glass, and see how close I am. A gentle reminder notification
would be nice. Save my progress on the phone.
Type to Claude, then Enter 
Build a packing-list app for trips: I make a list, check items off as I
pack, and can reuse the list next time. Keep it simple and friendly.

When Claude's ready, start the live preview and scan the QR code with your phone:

Run in Terminal 
npx expo start    # shows a QR code — scan it with your phone's camera

🎮 Games

The easiest games run right in the browser. Start a folder (mkproj my-game then claude) and ask for one:

Type to Claude, then Enter 
Build a memory match game in the browser: a grid of cards, I flip two at
a time to find matching pairs, with a moves counter and a timer. Fun,
colourful, and it celebrates when I win. Then start it so I can play.
Type to Claude, then Enter 
Make a quiz game about animals: 10 multiple-choice questions, a running
score, and a friendly results screen at the end with a "play again"
button. Mobile-friendly.
Type to Claude, then Enter 
Build a classic Snake game I can play with the arrow keys (and swipes on
a phone): it speeds up as I score, and shows my high score.

Want a “real” downloadable game with characters and physics? Start from the game template — it sets up Godot, a free game engine:

Run in Terminal 
launchpad new      # choose "game" when it asks
Type to Claude, then Enter 
Help me build a simple 2D platformer in Godot: a character that runs and
jumps, a few platforms, coins to collect, and a score. Walk me through it
step by step and explain what each part does.

Add a feature

Run these inside an existing project — dev, then cd your-project, then claude.

User accounts (sign up & log in)

Add user accounts to this app: sign up, log in, log out, and a "you must
be logged in" guard on the main page. Use Supabase for this. Walk me
through making the free Supabase account and where to paste the keys, keep
them in .env.local, and add a couple of tests.

Shortcut: the assistant can run launchpad add supabase to scaffold this for you after you sign in.

Save data in a database

I want to store "projects" for each user (a title, a description, and a
date). Add the database table, the code to create / read / update / delete
them, and a simple page to manage them. Explain in plain English what you
set up.

Take payments

Add a paid plan with Stripe: a "Subscribe" button, Stripe Checkout, and a
premium page that unlocks after payment. Use Stripe TEST mode, walk me
through getting my test keys, never commit them, and add a webhook to
confirm payment.

Shortcut: launchpad add stripe scaffolds the test-mode Checkout + webhook once you've signed in to Stripe. Always start in test mode — no real money moves.

Let people upload a photo

Let users upload a profile photo and show it. Store the file safely, limit
it to images under 5 MB, and handle the "upload failed" case nicely.

Send emails or notifications

When someone submits the contact form, email me the message. Walk me
through setting up a free email-sending service and keep the key in
.env.local.

Add an AI feature

Add a "summarise this" button that sends the text to Claude and shows a
short summary. Use the Anthropic API with the latest Claude model, keep my
API key in .env.local, and handle errors gracefully.

The small stuff

Add a search box that filters the list as I type.
Add a dark-mode toggle that remembers my choice.

Make it look good

Can't judge design yourself? That's fine — ask the assistant to hold the bar for you.

Give this a clean, modern visual design — good spacing, a nice font
pairing, a calm colour palette, and a clear visual hierarchy. Use your
design skills and show me a before / after.
This feels a bit plain. Add some polish: subtle hover effects, a little
motion when things appear, and a friendly "empty state" for when there's
no data yet.
Make this look great on a phone: stack the layout, bigger tap targets, and
no sideways scrolling. Then open it at a phone size and show me a
screenshot.
Run an accessibility check on the main pages and fix what it finds —
labels, colour contrast, and keyboard navigation — so everyone can use it.

When you're stuck or confused

This is the most useful section on the page. Building means hitting little snags — that's normal, not a sign you did something wrong. Here's exactly what to say for each one.

Something broke (you saw red text or an error)

First and simplest: copy the exact red text (or take a screenshot of it), paste it to the assistant, and add:

This is the error I'm seeing. Explain in plain English what went wrong,
fix it, and add a test so it can't happen again.
📋

How to copy the error: drag to select the red text in the Terminal (or the browser), then press ⌘+C. Paste into the chat with ⌘+V. A screenshot works too — press ⌘+Shift+4, drag a box, and the picture saves to your Desktop.

You don't understand what it did — or a word it used

Never just nod along. Ask:

Explain what you just changed like I'm brand new to this — no jargon, and
tell me why it matters.
What does "" mean, in simple terms? Give me a
real-world comparison.

It's not what you pictured

Don't settle. Describe the gap between what you got and what you wanted:

That's close, but not what I meant. I wanted the buttons across the top,
not down the side, and the colours warmer. Try again.

Undo — go back to how it was

Every checkpoint you saved is a point you can return to. Just say:

That last change made things worse — undo it and go back to the previous
checkpoint.

This is exactly why we save checkpoints often — they're your safety net.

It worked yesterday, but not today

If you turned on error tracking, let it find the cause:

Something's broken that worked before. Check Sentry for the latest error,
explain what went wrong, and fix it.

No error tracking yet? Turn it on — it captures the exact file and line when something fails.

You're completely stuck and want a human

Make a tidy, secret-free report you can hand to a friend (or paste into a forum):

Run in Terminal 
launchpad report     # writes a shareable report; scans itself for secrets first

Still nothing? The Help page covers the handful of issues that occasionally need you.

Test it (so it stays working)

A test is a tiny program that checks your app still works — so a change next week doesn't quietly break something today.

Write tests for what we just built — quick checks for the logic and one
end-to-end test for the main flow — then run them and show me they pass.

Save, undo & back up

A checkpoint (a “commit”) is a save point you can always return to. Saving often is what makes experimenting safe.

Save this as a checkpoint with a clear message, then back it up to my
private GitHub.
That made it worse — undo it and go back to the previous checkpoint.
☁️

Why back up to GitHub? A checkpoint lives on your Mac. Pushing it to your private GitHub repo puts a copy safely online, so your work survives even if your Mac doesn't. It's private by default — only you can see it.

Put it online (share a link)

Deploy this so I can share a link. Use Vercel (free to start), connect my
GitHub repo, and give me the URL. Tell me exactly what's now public.

Shortcut: launchpad add vercel links the project, securely uploads your keys to Vercel's private settings (not to GitHub), and deploys.

For an instant share of a single page, image, or PDF — no account needed:

Publish this page to here.now and give me the link.
🌍

Published means public. Anyone with the link can open it, and anonymous here.now links stop working after 24 hours. For anything private, ask for a password-protected site, and never publish secrets.

See your data

Add a few realistic example rows to my database so I have something to
look at, and tell me how to open Beekeeper Studio to see them.

Then open it yourself any time — Beekeeper Studio is a friendly point-and-click app (a “GUI”) for browsing your database:

Run in Terminal 
open -a "Beekeeper Studio"

Keep your keys safe

An API key is a secret password your app uses to talk to another service (like payments or email). It must never end up in your code or on GitHub. Two safe homes for it:

  • A .env.local file — a private file that Git is set to ignore, so it's never backed up or published. This is the default and it's fine.
  • 1Password — even safer: the key lives in your vault, and an assistant can run your app with it loaded into memory, never written to disk.
Run in Terminal 
launchpad secrets set STRIPE_SECRET_KEY   # store a key (in 1Password if signed in)
launchpad secrets run -- npm run dev      # run with secrets loaded, nothing on disk
launchpad secrets status                  # which mode am I in?

Not signed in to 1Password? Every command falls back to .env.local and tells you so. Turn 1Password on anytime with op signin.

🛡️

A safety net runs for you: a hook scans every checkpoint for secrets and blocks the save if it finds one. If that ever happens, it's doing its job — take the secret out of the code, put it in .env.local, and save again. Never get around it with --no-verify.

Turn on error tracking

When your app misbehaves, Sentry captures the exact error, file, and line — so “it's broken” becomes “here's precisely what failed.” Your app is already wired for it; you just turn it on:

Turn on Sentry error tracking for this app: create or find the project,
get the DSN, and set it up. Then throw a test error and show me it appears
in Sentry.

The assistant fetches the key and runs launchpad sentry-setup. No account yet? It walks you through the free signup, or you can paste a DSN from sentry.io into launchpad sentry-setup.
