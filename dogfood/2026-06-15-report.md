## Findings

### P1 — fixed
**F1. Codex CLI install is unreliable (npm dist-tag regression).**
The prior VM run's only red (`✘ codex on PATH`) was *"Could not find Codex package or platform npm release assets for Codex 0.139.0."* This is **not transient** — since Codex 0.100.0 the CLI ships platform binaries via per-platform npm dist-tags / optional deps, and `npm`-backed installs (including the native `chatgpt.com/codex/install.sh` script) intermittently fail to resolve the platform subpackage (openai/codex issues #11744, #12931, #17432).
*Fix:* `bootstrap.sh` now prefers `brew install --cask codex` (self-contained bottle, no npm dist-tags) and falls back to the native script. → being VM-validated.

**F2. The pre-commit `project-tests` hook blocks every non-Node project.**
The hook ran `npm test --if-present` with `always_run: true`. In a directory with **no `package.json`** (a static site, a Python project, `launchpad new blank`, `mkproj scratch`), `npm test` exits **254 (ENOENT)** → the hook fails → the **first commit (and every commit) is blocked**. Verified on host across 4 cases.
*Fix:* entry is now `bash -c 'if [ -f package.json ]; then npm test --if-present; fi'` — a clean no-op without `package.json`, still **failing the commit on a real test failure** (verified: bare=0, no-script=0, failing-test=1, passing-test=0). → being VM-validated.

### P2 — fixed
**F3. Web template used the deprecated `--no-eslint` flag.** Next.js 16's `create-next-app` renamed it to `--no-linter` (the old one works only via runtime back-compat). Also `--agents-md` now defaults **on**, writing a `CLAUDE.md`/`AGENTS.md` into every scaffolded project (which would shadow the global house-rules).
*Fix:* scaffold now passes `--no-linter --no-agents-md`. → being VM-validated.

### P2 — documented (low value / not fixed)
**F4. Expo template copy says "tabs".** Live check: bare `create-expo-app@latest` scaffolds **SDK 54** (one behind stable **56**) and the default template is Expo Router + TypeScript **without** the tabs layout. The starter still runs key-free (the important property). *Recommendation:* soften the "tabs" wording in `templates/README.md`; optionally pass `--template tabs` for a richer starter. Cosmetic.

**F5. `replace_managed_block` re-appends a re-run block at end-of-file.** Re-running an *earlier* module standalone (e.g. `bash modules/01-shell.sh`, which the troubleshooting docs suggest) moves its `~/.zshrc` block below later blocks and can add a blank line. **No duplication** (block counts stayed 1), functionally harmless (zsh reads the whole file), and the normal sequential install + `update.sh` are byte-stable. Left as-is — reworking the core helper to preserve block position is higher risk than the cosmetic reward.

### Contested — NOT changed (flagged for binary-level re-verification)
A drift agent claimed three "fixes" that **contradict the repo's prior direct-binary/VM verification** (and this stack's green doctor runs). Per the repo's own principle — *verify against the installed binary, not general web docs* — these were **not applied**:
- *"Antigravity rules path `~/.gemini/AGENTS.md` is wrong → `.antigravity.md`"* — repo verified against the agy v1.0.8 binary and symlinks `AGENTS.md` **and** `GEMINI.md`; doctor's Antigravity house-rules check is green in-VM.
- *"Skills global path `~/.agents/skills` is wrong → per-agent dirs"* — a prior probe-VM empirically confirmed `~/.agents/skills` is the read path; all skills checks are green in-VM.
- *"Drop the Superpowers `enabledPlugins` settings.json step"* — doctor greps that marker and is green; removing it would break the check.
If revisiting, re-verify against the installed binaries, not docs.

## Gaps & innovations (ranked by impact × effort)

**★ TOP: Wire up the already-installed 1Password CLI for secret management.**
`00-foundation.sh` installs `1password-cli` (`op`) but **nothing uses it** (zero references repo-wide). Meanwhile "manage your API keys safely" is the #1 friction *and* risk for a non-technical user, and the whole stack's answer today is "paste it into `.env.local` by hand." Wiring `op` (an `AGENTS.md` house-rule + a `launchpad secrets` helper to store/inject keys via `op run` / `op inject`) would let agents reference secrets without them ever touching plaintext — directly serving the safety mission with a tool that's *already paid for and installed*. **High impact, medium effort.**

| Idea | Impact | Effort | Why |
|---|---|---|---|
| **1Password secret injection** (above) | ★★★ | ●● | installed-but-unused; kills the biggest friction+risk |
| **Auto-provision the Sentry DSN** via the Sentry MCP (agent creates the project + writes the DSN to `.env.local`) | ★★★ | ●● | closes the "errors auto-flow" loop; today the DSN is a manual paste |
| **`launchpad status` dashboard** — per-project backup/test/deploy/running state (ties git + gh + ports + Sentry) | ★★ | ●● | no overview today; surfaces unbacked-up work |
| **Pre-warm pre-commit hook envs** during install (or message "setting up safety checks ~1 min") | ★★ | ● | first commit silently downloads gitleaks/biome envs — reads as a hang |
| **Guided provisioning wizards** for Supabase/Stripe/Vercel (turn the recipes from prompts into account-creation + key-injection flows) | ★★★ | ●●● | login/payments are the most-wanted features and the hardest first step |
| **Spend guardrail** — daily `ccusage` check that notifies over a threshold | ★★ | ● | 3 full-autonomy agents can run up cost unseen |
| **`launchpad doctor --fix`** non-agent self-heal (re-run failing modules) | ★ | ●● | resilience when no agent is driving |
| **Backup nudge** — warn on uncommitted/unpushed work | ★★ | ● | "private backup" is only as good as the last push |
| **Onboarding/sign-in checklist** (`launchpad signin` walks GitHub + Sentry + here.now per agent) | ★★ | ●● | per-agent OAuth ×3 services is confusing |

**Secret-scanning nuance (informational, not a bug):** gitleaks correctly **ignores** low-entropy / well-known *example* secrets (the AWS docs key `AKIAIOSFODNN7EXAMPLE`, a sequential `ghp_0123…`) to avoid false positives, and **catches** realistic high-entropy ones (verified live: `github-pat`, `stripe-access-token`). A real pasted key is protected; an obvious placeholder won't trip the gate — worth knowing when demoing.

**Test-harness notes (my dogfood scripts, *not* the product):** the first probe pass used GNU `timeout` (absent on macOS) and allow-listed/low-entropy fixtures — both corrected; `vm-probes.sh` also mis-exits 1 when zero probes fail (`grep -c` under `set -e`). All cosmetic; the product results above are clean.
